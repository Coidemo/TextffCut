# テキスト処理フローのリファクタリング前後の比較

## 1. はじめに

このドキュメントでは、TextffCutアプリケーションにおける「切り抜き箇所のテキスト処理」の仕組みが、システムの再設計（リファクタリング）前後でどのように変化したかを、**実際のコードを確認した上で**説明します。

本ドキュメントはコミット `afcdfd79` のコードと現在のコードを比較して作成されています。

### 1.1 テキスト処理とは何か？

ユーザーが動画から切り抜きたい部分を指定する際、以下の処理が行われます：

1. **文字起こし結果の確認**: 動画の音声を文字に変換した結果を表示
2. **切り抜き箇所の入力**: ユーザーが切り抜きたい部分のテキストを入力
3. **差分の検出**: 入力されたテキストが元の文字起こし結果とどう違うかを確認
4. **時間の特定**: テキストに対応する動画の時間を特定
5. **エラーチェック**: 問題がないか確認し、あれば警告を表示

## 2. リファクタリング前の処理フロー

### 2.1 全体の流れ

```
[ユーザー] → [テキスト入力] → [更新ボタンクリック] → [処理開始]
                                                          ↓
                                                    [テキストの前処理]
                                                          ↓
                                                    [差分検出]
                                                          ↓
                                                [has_additions()チェック]
                                                      ↙            ↘
                                            [追加文字なし]    [追加文字あり]
                                                ↓                  ↓
                                          [時間計算]    [エラー表示・削除ボタン]
```

### 2.2 各処理の詳細

#### 入力データ
- **元の文字起こしテキスト**: 動画から自動生成されたテキスト全文
- **編集されたテキスト**: ユーザーが入力した切り抜き箇所のテキスト

#### 処理の流れ（コードの実際の流れ）

1. **テキストの前処理**
   - 境界調整マーカー（`[<`, `[>`, `<]`, `>]`）の存在を確認
   - マーカーがある場合は `remove_boundary_markers()` で除去
   - 区切り文字（`---`, `——`, `－－－`）の検出

2. **差分検出（`TextProcessor.find_differences()`）**
   ```python
   diff = text_processor.find_differences(
       full_text, cleaned_text, skip_normalization=has_boundary_markers
   )
   ```
   - `SequenceMatcher` を使用して差分を計算
   - 共通部分（`common_positions`）を `TextPosition` のリストとして保存
   - 追加文字（`added_chars`）を `set` として保存
   - 追加文字の位置（`added_positions`）も記録

3. **結果の判定（`diff.has_additions()` メソッド）**
   ```python
   def has_additions(self) -> bool:
       """追加された文字があるか"""
       return len(self.added_chars) > 0
   ```
   - **追加文字がない場合**: 
     - `diff.get_time_ranges(transcription)` で時間計算
     - セッション状態に保存して処理継続
   - **追加文字がある場合**:
     - `st.session_state.show_error_and_delete = True` を設定
     - エラーメッセージと削除ボタンを表示
     - 処理を中断

#### 出力データ
- **正常時**: 
  - 時間範囲のリスト `[(start1, end1), (start2, end2), ...]`
  - セッション状態に `time_ranges` として保存
- **エラー時**: 
  - エラーメッセージ: "元の動画に存在しない部分が含まれています"
  - 削除ボタンの表示（追加文字を削除してテキストエディタを更新）
  - モーダル表示（`show_red_highlight_modal()`）で追加文字を赤でハイライト

### 2.3 特徴
- **シンプルで直線的な処理フロー**: `main.py` の更新ボタン処理内で完結
- **エラー時は即座に警告表示**: `has_additions()` チェック後すぐに表示
- **全ての処理が1つの場所に集約**: `TextProcessor` クラスに集中
- **データ構造が明確**: `TextDifference` に `added_chars` フィールドが存在

## 3. リファクタリング後の処理フロー

### 3.1 全体の流れ

```
[ユーザー] → [View層] → [Presenter層] → [Gateway層] → [Core処理]
    ↑           ↓            ↓              ↓              ↓
    └───────────┴────────────┴──────────────┴──────────────┘
         結果の表示      ビジネスロジック   データ変換    実際の計算
```

### 3.2 層（レイヤー）の役割

1. **View層（表示層）**
   - ユーザーインターフェースの管理
   - ボタンクリックなどのイベント処理
   - 結果の表示（正常時の表示、エラーモーダルなど）

2. **Presenter層（制御層）**
   - ビジネスロジックの実行
   - データの妥当性チェック
   - 処理フローの制御

3. **Gateway層（変換層）**
   - 新旧システム間のデータ変換
   - インターフェースの統一

4. **Core層（中核処理層）**
   - 実際の差分検出アルゴリズム
   - 時間計算処理

### 3.3 処理の詳細な流れ

#### ステップ1: ユーザー操作（View層）
- **入力**: ユーザーのボタンクリック
- **処理**: イベントをPresenter層に通知
- **出力**: Presenterへの処理依頼

#### ステップ2: ビジネスロジック実行（Presenter層）
- **入力**: 編集されたテキスト
- **処理**:
  1. 境界調整マーカーの検出
  2. 区切り文字（---）の検出
  3. 適切な処理方法の選択
- **出力**: Gateway層への差分検出依頼

#### ステップ3: データ変換と差分検出（Gateway層 → Core層）
- **入力**: 元のテキスト、編集されたテキスト
- **処理**:
  1. 旧システムの `TextProcessor.find_differences()` を呼び出し
  2. 結果を新しいドメインエンティティ `TextDifference` に変換
     - **問題**: 旧システムの `added_chars` フィールドが変換されない
     - `TextConverter.from_legacy()` で `common_positions` のみ変換
- **出力**: 差分情報（`added_chars` 情報が欠落）

#### ステップ4: 結果の処理（Presenter層）
- **入力**: 差分情報（ドメインエンティティ）
- **処理**:
  1. **追加文字の再検出が必要**（元の `added_chars` が失われたため）
     ```python
     # differencesリストから再度検出
     for diff_type, text, _ in result.differences:
         if diff_type == DifferenceType.ADDED:
             added_texts.append(text)
     ```
  2. 追加文字がある場合は時間計算をスキップ（エラーは設定しない）
  3. ない場合は `get_time_ranges()` で時間計算
- **出力**: ViewModelへの状態更新

#### ステップ5: 表示更新（View層）
- **入力**: ViewModelの状態
- **処理**:
  1. 更新ボタン処理で `_has_added_chars()` メソッドを呼び出し
     ```python
     def _has_added_chars(self, differences) -> bool:
         if hasattr(differences, "differences"):
             for diff_type, _, _ in differences.differences:
                 if diff_type == DifferenceType.ADDED:
                     return True
         return False
     ```
  2. 追加文字があれば `show_modal = True` を設定してモーダル表示
  3. なければ正常処理の結果を表示
- **出力**: ユーザーへの画面表示

## 4. 主な変更点と問題点

### 4.1 変更点

1. **処理の分散**
   - 前：`main.py` と `TextProcessor` で完結
   - 後：View → Presenter → Gateway → Core の4層に分散

2. **データの変換**
   - 前：`TextDifference` データクラスを直接使用
   - 後：Legacy → Domain → ViewModel の変換が発生

3. **エラーチェックの複雑化**
   - 前：`diff.has_additions()` の1回のチェック
   - 後：Presenter層とView層で2回チェック（情報欠落のため）

4. **追加文字の扱い**
   - 前：`added_chars: set[str]` フィールドで明確に管理
   - 後：`differences` リストから都度抽出が必要

### 4.2 現在の問題点

#### 問題1: 情報の欠落
- **現象**: 差分検出で見つかった「追加文字」の情報が、データ変換時に失われる
- **影響**: 後続の処理で追加文字を再度探す必要がある
- **原因**: 
  - 旧システム: `TextDifference` に `added_chars: set[str]` フィールドあり
  - 新システム: ドメインエンティティに `added_chars` フィールドなし
  - 変換処理: `TextConverter.from_legacy()` が `added_chars` を変換しない

#### 問題2: 過度な処理の中断（変わらない）
- **現象**: 句読点1つでも追加されていると、全体の処理が止まる
- **例**: 
  - 入力テキスト：「今日は晴れです。」（最後の句点が追加）
  - 元のテキスト：「今日は晴れです」
  - 結果：全体がエラー扱いになる（本来は「今日は晴れです」の部分は処理可能）
- **影響**: ユーザーは小さな違いでも修正を強制される
- **注**: この動作はリファクタリング前後で変わっていない

#### 問題3: 処理フローの複雑化による保守性の低下
- **現象**: 同じチェックを複数箇所で行う必要がある
- **実際のコード**:
  - Presenter層: `differences` リストから `DifferenceType.ADDED` を探す
  - View層: 同じく `differences` リストから `DifferenceType.ADDED` を探す
- **影響**: 
  - バグが発生しやすい（今回のように一部の実装が漏れる）
  - 修正時に複数箇所の変更が必要

## 5. バズクリップ機能での具体例

### 5.1 バズクリップとは
AIが動画から「バズる（話題になる）」可能性の高い30〜40秒の部分を自動で選んでくれる機能です。

### 5.2 問題の発生例

1. **AIが生成したテキスト**:
   ```
   お金持ちとか外国人とかお金に余裕のある高齢者とかからも
   平等に取れて社会福祉に使われる消費税は僕は上げてもいいと
   すら思っていますね。
   ```
   （AIが句読点を追加）

2. **元の文字起こし**:
   ```
   お金持ちとか外国人とかお金に余裕のある高齢者とかからも
   平等に取れて社会福祉に使われる消費税は僕は上げてもいいと
   すら思っていますね
   ```
   （句読点なし）

3. **結果**:
   - 最後の「。」だけが違う
   - リファクタリング前：
     - `diff.has_additions()` が `True` を返す
     - エラーメッセージ表示
     - 削除ボタンで句点を削除可能
   - リファクタリング後（バグ修正前）：
     - Presenter層で追加文字を検出できず
     - 時間計算時にエラー発生
     - 「タイムスタンプが見つかりません」と表示
   - リファクタリング後（バグ修正後）：
     - リファクタリング前と同じ動作に戻る

## 6. 改善の方向性

### 6.1 短期的な改善
1. **エラーメッセージの改善**
   - 「一部に文字起こし結果にない文字（。）が含まれています」
   - どの部分が問題か具体的に示す

2. **部分的な処理の継続**
   - 共通部分は正常に処理
   - 追加部分だけ警告またはスキップ

### 6.2 長期的な改善
1. **データ構造の見直し**
   - 追加文字の情報を適切に保持
   - 変換時の情報欠落を防ぐ

2. **処理フローの簡素化**
   - エラーチェックの一元化
   - ユーザーにとって分かりやすい流れに

## 7. まとめ

リファクタリングによってシステムの構造は整理されましたが、以下の課題が生じています：

1. **複雑性の増加**: 
   - 処理が4層に分散し、流れを追うのが困難
   - 同じチェックを複数箇所で実装する必要がある

2. **情報の欠落**: 
   - `added_chars` フィールドが新しいドメインエンティティに存在しない
   - データ変換時に重要な情報が失われる
   - 結果として、後続処理で再計算が必要

3. **保守性の問題**:
   - 今回のバグのように、一部の層での実装漏れが発生しやすい
   - 修正時に複数層の変更が必要

4. **性能への影響**:
   - 追加文字の検出を2回行う必要がある（無駄な処理）
   - データ変換のオーバーヘッド

これらの問題は、クリーンアーキテクチャの適用時によく見られる課題です。理想的には、ドメインエンティティを拡張して `added_chars` 情報を保持し、各層で一貫した処理を行うべきです。
