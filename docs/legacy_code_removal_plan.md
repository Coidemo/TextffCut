# レガシーコード削除計画

## 概要
クリーンアーキテクチャへの移行がほぼ完了したため、レガシーコードの削除計画を立案します。

## 削除対象ファイル一覧

### 1. メインエントリーポイント
- `main_legacy.py` - レガシー版のメインファイル
- 依存関係：servicesパッケージ、uiパッケージの旧実装

### 2. servicesパッケージ（レガシー版のみで使用）
- `services/transcription_service.py`
- `services/video_processing_service.py`
- `services/text_editing_service.py`
- `services/export_service.py`
- `services/configuration_service.py`
- `services/base.py`
- `services/__init__.py`

### 3. uiパッケージの旧実装（確認が必要）
- レガシー版でのみ使用されるコンポーネント
- 新アーキテクチャで移行済みのコンポーネント

### 4. coreパッケージ（Gatewayでラップ済み）
- `core/transcription.py` - TranscriptionGatewayでラップ済み
- `core/video.py` - VideoProcessorGatewayでラップ済み
- `core/text_processor.py` - TextProcessorGatewayでラップ済み
- `core/export.py` - 各種ExportGatewayでラップ済み

## 削除前の確認事項

### 1. 機能の完全性確認
- [ ] 文字起こし機能が正常に動作
- [ ] テキスト編集・差分検出が正常に動作
- [ ] 無音削除が正常に動作
- [ ] 各種エクスポート（FCPXML、SRT、動画）が正常に動作
- [ ] API/ローカル切り替えが正常に動作

### 2. テストの実行
- [ ] 単体テストがすべてパス
- [ ] 統合テストがすべてパス
- [ ] E2Eテストがすべてパス

### 3. 依存関係の確認
- [ ] main.pyがmain_legacy.pyの機能を完全にカバー
- [ ] すべてのUIコンポーネントが新アーキテクチャで動作
- [ ] 設定ファイルやキャッシュの互換性

## 段階的削除計画

### Phase 1: servicesパッケージの削除（優先度：高）
1. main.pyで全機能が動作することを確認
2. servicesパッケージを削除
3. main_legacy.pyが起動しないことを確認

### Phase 2: レガシーUIコンポーネントの削除（優先度：中）
1. uiパッケージ内のレガシー専用コンポーネントを特定
2. 新アーキテクチャで使用されていないことを確認
3. 段階的に削除

### Phase 3: coreパッケージのリファクタリング（優先度：低）
1. Gatewayでのみ使用されていることを確認
2. 必要に応じてGateway内に実装を移動
3. または、coreパッケージを内部実装として維持

### Phase 4: main_legacy.pyの削除（最終段階）
1. すべての機能がmain.pyで動作することを最終確認
2. ユーザーへの告知（必要に応じて）
3. main_legacy.pyを削除

## リスク管理

### 1. ロールバック計画
- Gitタグでレガシー版の最終バージョンを保存
- 削除前にブランチを作成

### 2. 移行期間
- 1-2週間の並行運用期間を設ける
- ユーザーフィードバックを収集

### 3. ドキュメント更新
- README.mdの更新
- 起動方法の変更を明記
- 移行ガイドの作成

## 実行タイミング

### 推奨される実行時期
1. **即座に実行可能**: servicesパッケージの削除
2. **1週間後**: レガシーUIコンポーネントの削除
3. **2週間後**: main_legacy.pyの削除

### 前提条件
- すべての機能テストがパス
- ユーザーからの重大なバグ報告がない
- 新アーキテクチャが安定して動作

## まとめ

クリーンアーキテクチャへの移行はほぼ完了しており、レガシーコードの削除準備が整っています。段階的な削除により、リスクを最小限に抑えながら、コードベースをクリーンに保つことができます。
