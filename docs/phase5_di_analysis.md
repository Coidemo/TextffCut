# Phase 5: 依存性注入（DI）分析と計画

## 現状分析

### 1. 依存関係管理の現状

#### グローバル設定パターン
- `config.py`の最後で`config = Config.from_env()`によるグローバルインスタンス
- 83ファイルが`from config import config`で直接参照
- main.py内で新しいConfigインスタンスを作成する問題箇所あり

#### サービス層の初期化
- BaseServiceクラスがコンストラクタでConfigを受け取る
- 必要な箇所で都度サービスインスタンスを作成
- 同じサービスが複数回インスタンス化される可能性

### 2. 主な課題

1. **グローバル依存の解消**
   - 大量のファイルがグローバルconfigに依存
   - テストが困難（モック化が難しい）

2. **インスタンス管理の混乱**
   - 統一されたインスタンス管理がない
   - サービスのライフサイクルが不明確

3. **設定の動的変更**
   - Streamlitのセッション状態との同期
   - 実行時の設定変更への対応

4. **ワーカープロセスとの連携**
   - worker_*.pyが独自にConfigを作成
   - サブプロセス内でのDI管理

5. **段階的移行の必要性**
   - 既存コードベースが大きい
   - 一度に全て変更するのは困難

## Python DIライブラリの評価

### 1. dependency-injector
**特徴:**
- 最も人気のあるPython DIライブラリ
- 豊富な機能（Provider、Factory、Singleton等）
- 設定管理機能内蔵
- 型ヒントサポート

**評価:**
- ✅ 機能が充実
- ✅ ドキュメントが豊富
- ✅ 実績がある
- ❌ 学習曲線がやや急
- ❌ 設定が複雑になりがち

### 2. injector
**特徴:**
- Google GuiceにインスパイアされたDIライブラリ
- シンプルなAPI
- デコレータベースの設定

**評価:**
- ✅ シンプルで理解しやすい
- ✅ 軽量
- ❌ 機能が限定的
- ❌ 設定管理機能なし

### 3. pinject
**特徴:**
- GoogleによるDIライブラリ
- 自動配線機能

**評価:**
- ✅ 自動配線が便利
- ❌ メンテナンスが活発でない
- ❌ Python 3のサポートが不完全

### 4. 手動実装（Pure Python）
**特徴:**
- 外部依存なし
- 完全な制御
- シンプルなファクトリーパターン

**評価:**
- ✅ 外部依存なし
- ✅ 完全な制御
- ✅ 段階的移行が容易
- ❌ 機能を自分で実装する必要

## 推奨アプローチ

### dependency-injectorを採用する理由

1. **成熟度と実績**
   - 最も広く使われているPython DIライブラリ
   - 安定性が高い

2. **段階的移行のサポート**
   - 既存コードと共存可能
   - Provider パターンで柔軟な設定

3. **設定管理統合**
   - Config管理機能が内蔵
   - 環境変数との統合が容易

4. **テストサポート**
   - テスト用のオーバーライド機能
   - モックの注入が簡単
