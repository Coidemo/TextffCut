# タイムライン編集機能 - 要件分析と設計検証

## 1. 現状分析

### 1.1 現在の実装状況
- ✅ テキストベースの編集機能（差分検出による切り抜き指定）
- ✅ 時間範囲の抽出（time_ranges）
- ✅ 無音削除機能
- ✅ 動画/XML/SRTエクスポート
- ❌ 視覚的なタイムライン表示
- ❌ 切り抜き位置の微調整機能
- ❌ プレビュー再生機能

### 1.2 現在のワークフロー
1. 文字起こし実行
2. テキスト編集で切り抜き部分を指定
3. 処理実行（切り抜きのみ or 切り抜き+無音削除）
4. 結果をエクスポート

**問題点**：
- 切り抜き位置が正確でない場合、やり直しが必要
- 視覚的に確認できないため、試行錯誤が必要
- 微調整ができない

## 2. タイムライン編集機能の要件詳細

### 2.1 機能要件（要件定義書より）

#### 2.1.1 タイムライン表示
- **セグメント表示**: 切り抜きに指定した各部分を個別のセグメントとして並列表示
- **切り抜き区間表示**: 各セグメントの開始/終了点をタイムライン上で表示
- **波形表示**: 音声レベルの可視化によるカット位置の判断支援

#### 2.1.2 微調整機能
- **数値入力調整**: 開始/終了時間を秒単位で直接入力して調整
- **フレーム単位調整**: ±1フレーム、±5フレーム、±30フレームボタンによる段階的調整
- **キーボードショートカット**: 矢印キーによるフレーム単位の微調整操作
- **プレビュー再生**: 調整箇所の前後を再生して確認

### 2.2 技術的要件

#### 2.2.1 データ構造
現在のtime_rangesフォーマット：
```python
time_ranges = [(start1, end1), (start2, end2), ...]  # 秒単位の浮動小数点
```

必要な拡張：
```python
@dataclass
class TimelineSegment:
    id: str                    # セグメント識別子
    start: float              # 開始時間（秒）
    end: float                # 終了時間（秒）
    text: str                 # 対応するテキスト
    waveform_data: list[float] # 波形データ（オプション）
    
    def duration(self) -> float:
        return self.end - self.start
    
    def to_frames(self, fps: float = 30.0) -> tuple[int, int]:
        """フレーム番号に変換"""
        return int(self.start * fps), int(self.end * fps)
    
    def adjust_start(self, delta: float):
        """開始時間を調整"""
        self.start = max(0, self.start + delta)
        
    def adjust_end(self, delta: float, max_duration: float):
        """終了時間を調整"""
        self.end = min(max_duration, self.end + delta)
```

#### 2.2.2 UI要件
- **Streamlit対応**: 既存のStreamlit UIフレームワークで実装
- **リアルタイム更新**: 調整結果を即座に反映
- **レスポンシブ**: 異なる画面サイズに対応

### 3.3 技術的課題と解決策

#### 3.3.1 波形データの生成
**課題**: 大きな動画ファイルの波形データ生成は時間がかかる
**解決策**:
- 低解像度での波形生成（1秒あたり10-20サンプル）
- バックグラウンド処理 + プログレスバー表示
- キャッシュ機能の実装

#### 3.3.2 プレビュー再生
**課題**: Streamlitでの動画/音声再生の制限
**解決策**:
- HTML5 audio要素を使用した再生
- 特定区間のみの音声ファイル生成（FFmpeg）
- st.audio()コンポーネントの活用

#### 3.3.3 リアルタイム更新
**課題**: Streamlitの再実行による状態リセット
**解決策**:
- st.session_stateでの状態管理
- 更新箇所のみの部分的な再レンダリング
- デバウンス処理による過剰な更新の抑制

## 6. リスクと対策

### 6.1 技術的リスク
- **リスク**: Streamlitの制限により高度なUIが実装できない
- **対策**: 段階的な実装、代替UI案の準備

### 6.2 パフォーマンスリスク
- **リスク**: 大きな動画での波形生成が遅い
- **対策**: 非同期処理、キャッシュ、段階的読み込み

### 6.3 ユーザビリティリスク
- **リスク**: UIが複雑になりすぎる
- **対策**: シンプルなMVPから開始、ユーザーフィードバックの収集
