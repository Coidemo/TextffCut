# TextffCut クリーンアーキテクチャ基本設計書

## 1. 概要

### 1.1 この設計書の目的
TextffCutをより保守しやすく、拡張しやすく、テストしやすいシステムに作り変えるための設計方針を示します。

### 1.2 基本的な考え方
このシステムは「層（レイヤー）」という考え方で整理されています。ちょうど建物の階層のように、それぞれの層には明確な役割があり、上の層は下の層を使いますが、下の層は上の層のことを知りません。

**主な設計原則：**
- **単一責任**: 各部品（クラス）は一つの仕事だけを担当します
- **依存関係の管理**: 上位の機能は下位の機能を使いますが、逆はありません
- **変更への対応**: 新機能を追加しやすく、既存機能の修正が他に影響しにくい設計
- **部品の入れ替え**: 各部品は独立しているため、必要に応じて入れ替えられます
- **関心の分離**: 画面表示、業務ロジック、データ保存などを別々に管理します

## 2. システムの全体構造

### 2.1 層構造の概要
システムは4つの層に分かれており、上から下へ以下の順番で並んでいます：

1. **プレゼンテーション層**（画面表示）
2. **アプリケーション層**（業務処理）
3. **ドメイン層**（中核となるデータ構造）
4. **インフラストラクチャ層**（外部システムとの接続）

### 2.2 視覚的な構造図

```
┌─────────────────────────────────────────────────────────────┐
│          プレゼンテーション層（Presentation Layer）            │
│                    ユーザーが見て操作する部分                     │
│  ┌─────────────┐  ┌──────────────┐  ┌─────────────────┐   │
│  │    画面     │  │   画面状態    │  │  画面制御係    │   │
│  │  (Views)   │  │ (ViewModels) │  │ (Presenters)   │   │
│  └──────┬──────┘  └──────┬───────┘  └────────┬────────┘   │
│         │                 │                     │            │
├─────────┼─────────────────┼─────────────────────┼────────────┤
│         │                 │                     ▼            │
│         │                 │      アプリケーション層            │
│         │                 └────►  業務処理を行う部分           │
│         │                        (Use Cases)                │
│         │                           ┬                       │
├─────────┼──────────────────────────────┼────────────────────┤
│         │                              ▼                    │
│         │                      ドメイン層                    │
│         │                 データの基本構造を定義              │
│         │        ┌────────────────┐   ┌─────────────────┐ │
│         │        │ エンティティ     │   │  値オブジェクト   │ │
│         │        │ (主要データ)    │   │  (補助データ)    │ │
│         │        └────────────────┘   └─────────────────┘ │
├─────────┼───────────────────────────────────────────────────┤
│         │              インフラストラクチャ層                  │
│         └──────────► 外部システムとの接続を担当               │
│                    ┌─────────────┐  ┌─────────────────┐   │
│                    │ 既存コード   │  │   外部API       │   │
│                    │ との接続     │  │  との接続        │   │
│                    └─────────────┘  └─────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

### 2.3 データの流れ
1. ユーザーが画面を操作
2. 画面制御係が業務処理を呼び出し
3. 業務処理がデータを操作
4. 必要に応じて外部システムと通信
5. 結果を画面に表示

## 3. 各層の詳細説明

### 3.1 プレゼンテーション層
**役割**: ユーザーが直接触れる画面部分を管理します

#### 構成要素:
- **画面（Views）**: ユーザーが見るボタンや入力欄などの画面部品
- **画面状態（ViewModels）**: 画面に表示するデータを保持する箱
- **画面制御係（Presenters）**: ユーザーの操作を受けて、適切な処理を呼び出す司令塔

#### 動作の仕組み:
1. ユーザーがボタンをクリック
2. 画面制御係がそのクリックを受け取る
3. 必要な業務処理を呼び出す
4. 結果を画面状態に保存
5. 画面が自動的に更新される

### 3.2 アプリケーション層（業務処理）
**役割**: TextffCutの主要な機能を実現する処理を管理します

#### 主な業務処理:
- **動画の文字起こし処理**: 動画から文字を抽出する
- **テキスト差分検出処理**: 編集前後のテキストの違いを見つける
- **動画エクスポート処理**: 編集した動画を出力する
- **FCPXMLエクスポート処理**: 動画編集ソフト用のファイルを作成する

#### 重要なルール:
- 一つの処理は一つの仕事だけを行う（文字起こしなら文字起こしだけ）
- 画面のことは知らない（どんな画面から呼ばれても動作する）
- データの保存方法も知らない（保存は別の層に任せる）

### 3.3 ドメイン層（データ構造）
**役割**: システムで扱うデータの形を定義します

#### 主要なデータ（エンティティ）:
- **文字起こし結果**: 動画から抽出した文字情報
- **動画セグメント**: 動画の一部分を表すデータ
- **テキスト差分**: 編集前後の違いを表すデータ

#### 補助的なデータ（値オブジェクト）:
- **ファイルパス**: ファイルの場所を表す
- **時間範囲**: 開始時刻と終了時刻のセット
- **継続時間**: どれくらいの長さかを表す

#### なぜ分けるのか:
- 主要なデータは変更される可能性がある（例：文字起こし結果の修正）
- 補助的なデータは一度作ったら変更しない（例：10秒から20秒という時間範囲）

### 3.4 インフラストラクチャ層（外部接続）
**役割**: 外部のシステムやツールとの接続を担当します

#### 接続アダプター（つなぎ役）:
- **ファイル接続**: パソコン内のファイルを読み書きする（新規実装）
- **文字起こしAPI接続**: WhisperX連携（既存コードを活用）
- **動画処理接続**: FFmpeg操作（既存コードを活用）
- **無音検出接続**: 無音部分の検出（既存コードを活用）
- **エクスポート接続**: FCPXMLとEDL形式（既存コードを活用）、その他は新規実装

#### バランスの取れたアダプター戦略:
- **既存活用を増やした部分**:
  - FFmpeg操作（動画処理の核心）
  - WhisperX連携（外部API接続）
  - 無音検出（複雑なアルゴリズム）
  - FCPXML/EDLエクスポート（フォーマット仕様が複雑）
- **新規実装する部分**:
  - ファイル操作（シンプル）
  - SRTエクスポート（シンプルなテキスト形式）
  - UI/画面（完全刷新）
- **シンプルな設計**: 変換処理は最小限、将来の拡張性を確保

## 4. 部品の組み立て方（依存性注入）

### 4.1 なぜ部品を組み立てる仕組みが必要か
システムは多くの部品（クラス）から成り立っています。これらの部品を正しく組み合わせる必要があります。
例えば、文字起こし処理は、API接続部品を必要とします。この組み立てを自動化する仕組みが「依存性注入（DI）」です。

### 4.2 バランスの取れた部品管理構造
```
アプリケーション全体の管理箱（ApplicationContainer）
├── 設定情報（システム設定とセッションキー定義）
├── 接続部品の管理箱（GatewayContainer）
│   ├── ファイル接続部品（新規・設定不要）
│   ├── WhisperX接続部品（既存活用・設定必要）
│   ├── FFmpeg接続部品（既存活用・設定必要）
│   ├── 無音検出接続部品（既存活用・設定必要）
│   ├── FCPXMLエクスポート部品（既存活用・設定必要）
│   ├── EDLエクスポート部品（既存活用・設定必要）
│   └── SRTエクスポート部品（新規・設定不要）
├── 業務処理の管理箱（UseCaseContainer）
│   ├── 文字起こし処理
│   ├── テキスト編集処理
│   ├── 無音削除処理
│   └── エクスポート処理（形式別）
└── 画面部品の管理箱（PresentationContainer）
    ├── 画面状態（型安全な状態管理）
    ├── 画面制御係
    └── 画面表示
```

**簡素化のポイント**:
- サービス層を削除（不要な層を減らす）
- レガシー互換のための複雑な設定を削除
- 必要最小限の構造のみ

### 4.3 部品の組み立て手順
1. **システム起動時**: 
   - すべての部品を自動的に組み立てる
   - 必要な設定を各部品に渡す
   - 部品同士を正しくつなげる

2. **実行時の動き**: 
   - ユーザー操作 → 画面制御係 → 業務処理 → 接続部品 → 外部システム
   - 結果は逆順で戻ってくる

3. **重要なルール**:
   - 部品は必ず管理箱から取り出して使う
   - 勝手に新しい部品を作らない
   - 設定が必要な部品には必ず設定を渡す

## 5. データの流れ（実際の動作例）

### 5.1 文字起こし処理の流れ
```
1. ユーザーが「文字起こし開始」ボタンをクリック
                ↓
2. 画面が画面制御係に「ボタンが押された」と伝える
                ↓
3. 画面制御係が文字起こし処理を呼び出す
                ↓
4. 文字起こし処理がAPI接続部品を使ってWhisperXに依頼
                ↓
5. WhisperXが文字起こしを実行して結果を返す
                ↓
6. 結果が画面状態に保存される
                ↓
7. 画面が自動的に更新されて結果を表示
```

### 5.2 動画エクスポート処理の流れ
```
1. ユーザーが「エクスポート」ボタンをクリック
                ↓
2. 画面制御係がエクスポート処理を呼び出す
                ↓
3. エクスポート処理が動画処理接続部品を使用
                ↓
4. FFmpegが動画を処理してファイルを作成
                ↓
5. 完了通知が画面に表示される
```

## 6. エラー処理の仕組み

### 6.1 エラーの種類と対処
システムでは3種類のエラーを区別して処理します：

1. **データエラー**（例：開始時刻が終了時刻より後）
   - ユーザーに分かりやすいメッセージを表示
   - 「開始時刻は終了時刻より前に設定してください」

2. **処理エラー**（例：文字起こしが失敗）
   - 再試行の提案をする
   - 「文字起こしに失敗しました。もう一度お試しください」

3. **接続エラー**（例：インターネット接続が切れた）
   - 接続状況の確認を促す
   - 「ネットワーク接続を確認してください」

### 6.2 エラー処理の流れ
1. **エラー発生**: どこかの層でエラーが起きる
2. **エラー変換**: 技術的なエラーをユーザー向けメッセージに変換
3. **エラー記録**: 詳細情報をログに保存（開発者用）
4. **エラー表示**: ユーザーに分かりやすく表示

## 7. 設定の管理

### 7.1 設定の種類
システムには3つのレベルの設定があります：

1. **システム設定**
   - 動作モード（開発用/本番用）
   - 標準的な値（デフォルトの文字起こしモデルなど）

2. **接続設定**
   - APIキー（OpenAIなどのサービス用）
   - ファイルの保存場所
   - 外部ツールの設定

3. **ユーザー設定**
   - 画面の表示設定
   - 前回選択した値の記憶
   - 個人的な好み設定

### 7.2 設定の保存と読み込み
- 起動時に自動的に設定を読み込む
- 環境変数から重要な設定を取得（APIキーなど）
- ユーザー設定は自動的に保存される

## 8. テストの仕組み

### 8.1 テストの種類と目的

1. **部品単体のテスト**
   - 各部品が正しく動作するか確認
   - 例：文字起こし処理が正しく文字を抽出できるか

2. **部品連携のテスト**
   - 複数の部品が協力して動作するか確認
   - 例：画面制御係が業務処理を正しく呼び出せるか

3. **全体動作のテスト**
   - ユーザーの操作通りに動くか確認
   - 例：動画を選んで、文字起こしして、編集して、出力できるか

### 8.2 テストしやすい設計の利点
- 各部品が独立しているため、個別にテスト可能
- 外部システムを使わずにテスト用の代替品で確認できる
- 問題があった時、どの部品に原因があるかすぐ分かる

## 9. 移行戦略（改訂版）

### 9.1 レガシー対応を最小限にした移行計画

既存システムとの互換性は最小限に抑え、シンプルで保守しやすいシステムを目指します。

#### なぜレガシー対応を減らすのか
- **複雑性の削減**: 変換処理や二重管理が不要に
- **開発速度の向上**: 新機能の実装に集中できる
- **バグの減少**: シンプルな設計はバグが少ない
- **将来性の確保**: 技術的負債を作らない

### 9.2 現実的な段階的移行計画

#### Phase 1: 基盤構築とMVP（1.5週間）
- **Week 1-1.5**: アーキテクチャ基盤
  - DIコンテナのセットアップと学習
  - 基本的な層構造の実装
  - エラーハンドリングの統一化
  - 単体テストフレームワーク構築

#### Phase 2: 文字起こし機能（1週間）
- **Week 2**: 文字起こしの完全実装
  - WhisperXゲートウェイ（既存活用）
  - TranscriptionのMVPパターン実装
  - 基本的な動作確認とフィードバック収集
  - パフォーマンス測定（90分動画で10分以内）

#### Phase 3: テキスト編集機能（1週間）
- **Week 3**: 差分検出と編集
  - テキスト処理ゲートウェイ（既存活用を検討）
  - 編集UIの実装
  - 統合テストの実施

#### Phase 4: エクスポート機能（1.5週間）
- **Week 4-4.5**: 段階的なエクスポート実装
  - 無音検出ゲートウェイ（既存活用）
  - FCPXMLエクスポート（既存活用）から開始
  - EDL、SRTを順次追加
  - 各形式の動作確認

#### Phase 5: 統合とリリース（1週間）
- **Week 5**: 最終調整
  - 全機能の統合テスト
  - パフォーマンス最適化
  - ユーザビリティテスト
  - 段階的なロールアウト計画

**合計期間**: 5週間（バッファ込み）

### 9.3 シンプルな設計の利点
- コードが理解しやすい
- テストが書きやすい
- 修正や機能追加が簡単
- 新しい開発者も参加しやすい

詳細な分析は[レガシー互換性分析書](./legacy_compatibility_analysis.md)を参照

## 10. 実装時の重要な注意点

### 10.1 部品の正しい取り出し方
部品を使う時は、必ず管理箱から取り出します：

**正しい方法**:
```
1. アプリケーション全体の管理箱を起動
2. その中から必要な管理箱を取り出す
3. 管理箱から部品を取り出して使う
```

**間違った方法**:
```
1. 管理箱を個別に作成する
2. 後から無理やりつなげようとする
（これはエラーになります）
```

### 10.2 接続部品の設計ルール
1. **設定は外から渡す**: 接続部品の中で設定を作らない
2. **統一したパターン**: すべての接続部品は同じ方法で作る
3. **管理箱で管理**: すべての部品の関係は管理箱が管理する

### 10.3 よくあるエラーと対処法

| エラーメッセージ | 原因 | 対処法 |
|--------------|------|--------|
| DependenciesContainerエラー | 管理箱の使い方が間違っている | 親の管理箱から取り出す |
| config引数が足りない | 設定の渡し忘れ | 管理箱で設定を注入する |
| 循環参照エラー | 部品同士が互いに依存している | 依存関係を整理する |

### 10.4 部品のライフサイクル

1. **起動順序**
   - 設定を読み込む
   - 接続部品を準備
   - 業務処理を準備
   - 画面部品を準備

2. **終了時の処理**
   - 使用中のリソースを解放
   - 接続を正しく切断
   - データを保存

## 13. リスク管理とロールバック計画

### 13.1 主要リスクと対策

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|----------|------|
| 新実装のバグ | 高 | 中 | 段階的リリース、十分なテスト |
| パフォーマンス劣化 | 中 | 低 | 事前測定、最適化バッファ |
| 既存活用部分の不具合 | 中 | 低 | ラッパーでの隔離 |

### 13.2 ロールバック手順
1. **即時切り戻し**: 旧main.pyへの切り替え（1分以内）
2. **データ移行**: セッションデータの変換スクリプト
3. **ユーザー通知**: 事前準備されたメッセージ

## 14. 簡素化による変更点のまとめ

### 11.1 削除したもの
- **レガシーデータ変換機能**: 複雑さの原因
- **サービス層**: 不要な抽象化
- **並行稼働の仕組み**: 移行を迅速に
- **過度な設定管理**: シンプルな設定のみ

### 11.2 簡素化したもの
- **Gateway Adapter**: 2パターンのみ（新規 or 最小限の既存活用）
- **データ構造**: レガシー形式を考慮しない
- **エラー処理**: シンプルで分かりやすいメッセージ
- **テスト**: 複雑な互換性テストが不要

### 14.3 期待される効果
- **開発期間**: 3週間→5週間（現実的な見積もり）
