# DaVinci ResolveのSRT字幕改行問題と回避策

## 問題の概要

TextffCutが出力するSRTファイルがDaVinci Resolveで正しく表示されない問題があります。

### 症状
- SRTファイル内で正しく改行されている字幕が、DaVinci Resolveで読み込むと1行になってしまう
- 複数行の文字数が合計でカウントされ、自動改行が間違った位置で発生する

### 根本原因
分析の結果、以下の問題が判明しました：

1. **SRT仕様違反**：TextffCutが出力するSRTファイルは、各字幕エントリ間に必須の空行を含んでいない（修正済み）
2. **改行コードの問題**：DaVinci ResolveはCRLF（Windows形式）の改行コードを期待している
   - LF（Mac/Unix形式）では行頭カウントが狂うという報告あり
   - CRLF形式にすることで改行が正しく認識される
3. **`<b>`タグの使用**：DaVinci Resolve自身が出力するSRTでは使用されている

## 回避策

### 1. プロジェクト設定での対処（推奨）

DaVinci Resolveのプロジェクト設定で以下を調整：
1. プロジェクト設定を開く
2. 「字幕」タブを選択
3. 「1行あたりの最大文字数」を大きな値（例：100）に設定
4. これにより自動改行を無効化し、SRTファイルの改行を優先させる

### 2. SRTファイルでの対処

#### A. HTMLタグで囲む（最も効果的）
DaVinci Resolveが出力するSRTファイルを分析した結果、`<b>`タグで字幕テキスト全体を囲むことで改行が正しく認識されることが判明しました。

```srt
1
00:00:00,000 --> 00:00:04,299
<b>6月5日の木曜日かな木曜日はい
8時でございます</b>
```

**注意**: 複数行にまたがる場合、全体を1つのタグで囲む必要があります。

#### B. `<br>`タグを使用（効果は限定的）
```srt
1
00:00:00,000 --> 00:00:04,299
6月5日の木曜日かな木曜日はい<br>8時でございます
```

#### C. 改行コードの変更（効果は限定的）
- Unix形式（LF）からWindows形式（CRLF）に変更
- またはその逆

### 3. 手動での修正（非推奨）
DaVinci Resolve内で各字幕の改行を手動で削除・再入力する。
※大量の字幕がある場合は非現実的

## 現在の実装状況

TextffCutでは以下の実装を行っています：
- 日本語の禁則処理を考慮した自然な改行位置の決定
- 指定された文字数での改行
- SRT仕様準拠（各エントリ後に空行）✓
- `<b>`タグでの囲み込み ✓

### 発見された問題
- **連番の不一致**：XMLファイルには連番が付くが、SRTファイルには付かない
- **改行問題の継続**：`<b>`タグを追加してもDaVinci Resolveで改行が認識されない

## 今後の対応予定

### 1. 緊急修正（実装済み）
- **字幕エントリ間の空行追加**：SRTEntryクラスの修正 ✓
- **`<b>`タグの追加**：実装済み ✓

### 2. 追加修正（実装済み/実装中）
- **連番の統一**：XMLファイルと同じ連番をSRTファイルにも適用 ✓
- **改行コードのCRLF化**：Windows形式の改行コードに変更（実装中）
  - LF（Unix形式）では行頭カウントが狂うという報告
  - CRLF形式により改行が正しく認識される
   
### 3. その他の改善
- 改行コード選択オプション（LF/CRLF）
- DaVinci Resolve用プリセットの提供
