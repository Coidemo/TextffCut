# キャッシュ問題の修正テスト手順

## テスト概要
キャッシュがある動画で新規文字起こしを実行した際に、意図せずキャッシュが読み込まれていた問題が修正されたことを確認します。

## テスト手順

### 1. アプリケーションの起動
```bash
streamlit run main.py
```

### 2. テスト準備
1. 左サイドバーで「ローカルファイル」を選択
2. キャッシュがある動画を選択（例：`合理性は人や国によって違うよねえ、という話.mp4`）

### 3. 修正の確認

#### 3.1 キャッシュがある状態の確認
- 「📝 文字起こし」セクションに「保存済みの文字起こし結果」のドロップダウンが表示されることを確認
- ドロップダウンは**何も選択されていない状態**（"キャッシュを選択してください"と表示）であることを確認

#### 3.2 新規文字起こしのテスト
1. キャッシュを選択せずに、下の方にある「🖥️ ローカルで文字起こしを実行する」ボタンをクリック
2. 文字起こしが開始され、プログレスバーが表示されることを確認
3. **重要**: キャッシュから即座に読み込まれるのではなく、実際に文字起こし処理が実行されることを確認
   - プログレスバーが0%から徐々に進む
   - 「文字起こし処理中...」などのメッセージが表示される
   - 処理に時間がかかる（数分程度）

#### 3.3 キャッシュ使用のテスト
1. ページをリロード（F5キー）
2. 同じ動画を選択
3. 「保存済みの文字起こし結果」ドロップダウンからキャッシュを選択
4. 「💾 選択した結果を使用」ボタンをクリック
5. 即座に結果が読み込まれることを確認

### 4. 期待される動作

✅ **修正後の正しい動作**:
- キャッシュがあっても、新規実行ボタンを押すと新規文字起こしが実行される
- キャッシュを使用するには明示的に選択して「選択した結果を使用」ボタンを押す必要がある

❌ **修正前の問題のある動作**:
- 新規実行ボタンを押してもキャッシュが自動的に読み込まれてしまう
- プログレスバーが一瞬で100%になる

## デバッグ情報の確認

ターミナルに以下のようなログが表示されます：

**新規実行時（正常）**:
```
INFO: TranscribeVideoUseCase: Starting transcription for: 動画名.mp4 with model: medium
INFO: TranscribeVideoUseCase: use_cache parameter: False
INFO: TranscriptionGatewayAdapter.transcribe - APIモード: False
```

**キャッシュ使用時**:
```
INFO: LoadTranscriptionCacheUseCase: Loading cache: model=medium
INFO: キャッシュファイルから読み込み: videos/動画名_TextffCut/transcriptions/medium.json
```

## トラブルシューティング

もし期待通りに動作しない場合は、以下を確認してください：

1. **ブランチの確認**
   ```bash
   git branch --show-current
   # fix/api-key-not-found が表示されることを確認
   ```

2. **最新のコミットが適用されているか確認**
   ```bash
   git log --oneline -3
   # 以下のコミットが表示されることを確認
   # 84fe290f fix: キャッシュが意図せず読み込まれる問題を徹底的に修正
   ```

3. **アプリケーションを再起動**
   - Ctrl+C でアプリケーションを停止
   - `streamlit run main.py` で再起動

## テスト結果の報告

テスト結果を教えてください：
- [ ] キャッシュがある動画で新規実行すると、実際に文字起こし処理が実行された
- [ ] キャッシュを明示的に選択した場合のみ、キャッシュから読み込まれた
- [ ] ドロップダウンのデフォルトは「キャッシュを選択してください」だった
